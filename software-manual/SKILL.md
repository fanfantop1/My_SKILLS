# 软件说明书编写技能

## 核心规则

1. **每次输出前必须阅读代码**。不得在未查阅源码的情况下输出任何技术描述。先定位相关文件，Read/Grep 确认实现细节，再动笔。
2. **回答简洁**。用最短的段落把功能说清楚，避免冗余修饰，不写流水账。
3. **使用中文表述**。专业术语保留英文原名（如类名、函数名、算法名），其余用中文。
4. **以文字表述为主，尽量不用表格**。用户明确要求表格时才使用。
5. **用户说"阅读代码，确认"时**，逐项核对代码与之前输出的描述是否一致，给出核实结果，指出不符之处并修正。
6. **用户说"换种说法"时**，保持内容不变，仅改变句式或措辞。
7. **用户说"精简"时**，压缩文字但不丢失关键信息。
8. **用户给出章节标题时**，先查阅代码中对应的实现，再写出该章节的说明内容。
9. **发现文档与代码不一致时**，明确指出差异点，让用户决定是修改文档还是修改代码。

## 工作流程

1. 用户给出待编写内容的名称或标题
2. 根据标题判断涉及的功能模块，Grep/Read 定位源码位置
3. 通读相关代码，理清实际执行流程
4. 用简洁中文输出说明内容
5. 等待用户反馈（确认、修改、换说法、精简等）
6. 收到"阅读代码，确认"时逐项核实

## 输出风格

- 陈述事实，不堆砌形容词
- 一个段落讲清一件事
- 涉及多个步骤时用序号列表，但每项一句话
- 参数、默认值、取值范围等细节只在必要时提及
